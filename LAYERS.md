# LAYERS — Navigation for working on Tex

> **If you are a Claude instance landing in this repo for the first time, read this file first.**
> It tells you everything you need to know to work on a specific architectural layer without breaking the others.

## The six layers

Tex is organized as six architectural layers. Each layer has its own working doc you should read **before making changes to that layer**:

| # | Layer | Working doc | What it does |
|---|---|---|---|
| 1 | **Discovery** | [`docs/layers/LAYER_1_DISCOVERY.md`](docs/layers/LAYER_1_DISCOVERY.md) | Inventory AI agents across customer platforms (OpenAI, Slack, AWS, GitHub, Microsoft, Salesforce, MCP) |
| 2 | **Identity** | [`docs/layers/LAYER_2_IDENTITY.md`](docs/layers/LAYER_2_IDENTITY.md) | Per-agent identity, capability, behavioral evaluation |
| 3 | **Monitoring** | [`docs/layers/LAYER_3_MONITORING.md`](docs/layers/LAYER_3_MONITORING.md) | Scheduler, alerts, presence tracking, drift, telemetry |
| 4 | **Execution Governance** | [`docs/layers/LAYER_4_EXECUTION_GOVERNANCE.md`](docs/layers/LAYER_4_EXECUTION_GOVERNANCE.md) | The PDP — every PERMIT/ABSTAIN/FORBID decision |
| 5 | **Evidence** | [`docs/layers/LAYER_5_EVIDENCE.md`](docs/layers/LAYER_5_EVIDENCE.md) | Hash-chained evidence, C2PA, Web Proofs, ZK provenance, TEE attestation |
| 6 | **Learning** | [`docs/layers/LAYER_6_LEARNING.md`](docs/layers/LAYER_6_LEARNING.md) | Outcome validation, reporter reputation, calibration with human approval |

## How to use this file

### If the user asks you to work on a layer ("improve Layer 5", "make our discovery state-of-the-art")

1. **Read the layer's working doc.** It lists every package, every important file, the current state (WIRED vs TEST_ONLY), and the priority improvement candidates.
2. **Stay inside the layer's packages** unless the work demands crossing. If you need to touch another layer, note it explicitly in your reply.
3. **Run the impact check before finishing:** `python scripts/layers.py --layer N` confirms which packages you're working with.

### If the user asks something layer-agnostic

You can ignore this file. Layer organization is only relevant when scoping work to a specific layer.

### If you don't know which layer a file belongs to

Every package's `__init__.py` has an `[Architecture: Layer N (Name)]` header in its docstring and a `__layer__` constant. Read the file or:

```bash
python -c "from tex.<package> import __layer__, __layer_kind__; print(__layer__, __layer_kind__)"
```

Or just look at the layer query output:

```bash
python scripts/layers.py
```

## Reading order for a new Claude

1. `LAYERS.md` (this file) — 30 seconds
2. `ARCHITECTURE.md` — 5 minutes — what Tex actually is, end-to-end
3. The specific `docs/layers/LAYER_N_*.md` for the layer you're working on
4. `audit/EXECUTIVE_SUMMARY.md` if the question is "what's the current state, what's broken"
5. `audit/orphans/ORPHAN_REGISTRY.md` if the question is "which code is actively wired"

## What's NOT here

- **No claim files.** This repo had ~30 historical claim files (CLAIMS_HISTORY.md, FRONTIER_*.md, etc.). They were drift, not truth. All deleted. Don't reintroduce that pattern.
- **No "vision" docs.** Aspirational descriptions decoupled from code are how we got the drift problem. Anything documented here must be verifiable from the source code by `grep`.
- **No commit-message archives.** Git history is for that.

## Editing this file

If a layer's scope changes (a package moves from one layer to another, a new package is added), update:

1. The `__layer__` constant in the relevant `__init__.py`
2. The `[Architecture: ...]` header in the same file's docstring
3. The corresponding `docs/layers/LAYER_N_*.md`
4. Optionally re-run `python scripts/layers.py` to confirm

Every other layer doc is generated by reading the source. Don't hand-maintain duplicate lists.
