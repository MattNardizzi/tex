# Forge target — the armed dare

This directory holds the **armed** forge challenge: a real Tex-signed evidence
bundle plus the published public key to pin against. The dare in
[`../CHALLENGE.md`](../CHALLENGE.md) points here.

## What's published

| File | What it is |
|---|---|
| `canonical_bundle.pq.jsonl` | **Headline bundle.** Real Tex decisions sealed with a composite **ML-DSA-65 + Ed25519** signature (post-quantum). |
| `PUBKEY_STATEMENT.json` | The **out-of-band pin** for the `.pq` bundle. |
| `canonical_bundle.ecdsa.jsonl` | **Verify-anywhere bundle.** The same verdict superset sealed with **ECDSA-P256**, verifiable with `pyca/cryptography` alone (no ML-DSA backend needed). |
| `PUBKEY_STATEMENT.ecdsa.json` | The out-of-band pin for the `.ecdsa` bundle. |

The matching **private keys are NOT here and never will be.** They are founder-held,
generated into `forge/_private_keys/` (gitignored). Their secrecy is the *only*
thing standing between an attacker and a forged-but-pinned record — that is the
whole point of the dare.

## The out-of-band-pin doctrine (read this first)

The pin is obtained from the **statement file**, never from the bundle. A verifier
that read the public key out of the record it is checking would accept any
self-consistent forgery: an attacker re-signs a mutated payload with their own
fresh key, embeds their own public key, and the signature "checks out." The
verifier (`tex.bench.forge_target.load_published_pin`) therefore reads the pin
**only** from `PUBKEY_STATEMENT.json` and has **no fallback** to a bundle-embedded
key.

Even the in-repo statement file is not the final root of trust. A hostile fork
could swap *both* the bundle *and* the in-repo pin self-consistently. So the
statement carries a `public_key_sha256` **fingerprint**, and the founder ALSO
posts that fingerprint in a **second channel** (live site / signed git tag /
tweet). Cross-check the fingerprint you see here against that external copy — the
external one is canonical. (This is an operational guarantee, not a code one, and
is labeled as such.)

## Verify it yourself

```bash
# headline composite-PQ bundle (needs the ML-DSA backend, which
# `pip install -r requirements.txt` provides via cryptography>=48):
python scripts/verify_it_yourself.py --forge-target

# verify-anywhere classical bundle (pyca/cryptography alone):
python scripts/verify_it_yourself.py --forge-target --ecdsa
```

Or directly:

```bash
PYTHONPATH=src python -m tex.bench.forge_target \
    forge/canonical_bundle.pq.jsonl forge/PUBKEY_STATEMENT.json
```

Exit code `0` means the bundle verified **VALID** against the published pin:
chain intact, every signature self-verifies, every `payload_json` is in canonical
form, and every record is signed by the pinned key.

## Honest algorithm labeling

- The `.pq` bundle is **composite-ml-dsa-65-ed25519** — verification is
  **RUNTIME-DEPENDENT**: it requires an ML-DSA backend. A normal
  `pip install -r requirements.txt` provides it (`cryptography>=48` ships native
  ML-DSA via OpenSSL ≥3.5). A host *without* that backend cannot verify the
  composite signature and must use the `.ecdsa` bundle instead.
- The `.ecdsa` bundle is **ecdsa-p256** — verifiable with `pyca/cryptography`
  alone, on any host. It is the classical fallback, not a post-quantum claim.

Neither is a silent swap: both bundles are published, each with its own pin.

## Canonical-bytes gate

The verifier enforces a **canonical-bytes** invariant: each record's stored
`payload_json` must be byte-identical to its one canonical (sorted-key, compact)
serialization, with no duplicate keys. This is presentation integrity — it pins
the exact bytes a human reads to the one form Tex emits. The embedded signature
already binds the *meaning* (a meaning change without the private key fails
self-verification); the gate additionally closes the "two byte-strings, one
signature" presentation ambiguity.

## What's in the bundle (the verdict superset)

Both bundles contain the same superset of **real** Tex decisions, driven through
the live policy decision point (never hand-stuffed):

- a **structural FORBID** (a refund issued before identity was confirmed — the
  block lives in the action graph, invariant to wording),
- a **contract FORBID** (an OpenAI-style secret in outbound content),
- a benign **PERMIT** (an ordinary shipment confirmation),
- an ambiguous **ABSTAIN** (a wire transfer to an unknown recipient — the
  machine's own sealed refusal to decide: *signed silence*),
- a sealed human **refused** record (a named human's `no` on the held ABSTAIN,
  hash-chained to it).

This superset is deliberate so a future frontier-lock dare can pick its headline
from a known menu — signed silence / sealed ABSTAIN / sealed refusal / structural
FORBID — without re-arming.

## Re-arming

The artifacts here are generated by [`../scripts/arm_forge_challenge.py`](../scripts/arm_forge_challenge.py).
It refuses to overwrite committed artifacts without `--force`. Re-running mints
**new** keys and replaces the published target, so only do it deliberately.
