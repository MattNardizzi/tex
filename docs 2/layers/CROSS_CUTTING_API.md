# Cross-cutting — HTTP / API

> **Working doc.** The 22 routers spanning all six layers.

## What this concern covers

Every HTTP endpoint Tex exposes. Each route is a thin wrapper that:
1. Validates input against a Pydantic schema (`api/schemas.py`)
2. Authenticates the caller (when `TEX_REQUIRE_AUTH=1`)
3. Resolves the tenant context
4. Calls into one of the six layers
5. Returns a structured response

This is the **only** public surface customers interact with. Internal layers MUST NOT be reachable except through these routes.

## Key shared infrastructure

- `src/tex/api/auth.py` — API-key authentication, optional. Activated by `TEX_REQUIRE_AUTH=1`. Three header forms accepted: `X-Tex-Api-Key`, `Authorization: Bearer`, query string (dev only).
- `src/tex/api/schemas.py` — 1,294 lines of Pydantic request/response models. Single source of truth for the HTTP contract.
- `src/tex/api/dependencies.py` — FastAPI dependency providers (resolved at request time)
- `src/tex/api/error_handlers.py` — uniform error envelopes

## Adapter routes (gateway integration)

`guardrail_routes.py` includes adapters that translate gateway-native shapes into the canonical `/v1/guardrail` shape:

- Portkey
- LiteLLM
- Cloudflare
- Solo / Gloo
- TrueFoundry
- AWS Bedrock
- Microsoft Copilot Studio
- OpenAI AgentKit

Adding a new gateway = adding a new adapter route. Pattern is standardized.

## Improvement vectors

### 1. Wire MCP gate in front of MCP server (high impact, low effort)
30 lines in `mcp_server.py` to route every `tools/call` through `governance/kernel_mcp/SyscallGate`.

### 2. OpenAPI completeness review
Every route should have:
- `summary` and `description`
- All Pydantic models documented
- Example requests/responses
- Documented error codes

A sweep would improve developer experience for partners integrating against the API.

### 3. Rate limiting per tenant (medium impact, medium effort)
Today no per-tenant rate limit. Adding token-bucket per-tenant keeps a noisy tenant from degrading service for others.

### 4. Streaming responses for more routes
`/v1/guardrail/stream` exists. Adding SSE/streaming for `/v1/evaluate`, `/v1/discovery/scan`, and the evidence-bundle export would improve UX for long-running operations.

### 5. Webhook delivery infrastructure
Today most outputs are pull (customer polls). Push notifications via webhooks for:
- Discovery findings (new agent appeared)
- Drift alerts
- Policy activation events
- FORBID decisions
Webhook signing, retry, dead-letter handling, etc. — a real feature.

### 6. GraphQL surface (low priority)
Some enterprise integrators want GraphQL. Today REST only. Strawberry on top of the existing Pydantic schemas is a few days of work; only worth doing on customer demand.

## Constraints

- **Every route is layered**: HTTP route → Pydantic validation → command (in `commands/`) → layer code. Never call layer code directly from the route.
- **Tenant ID must be resolved on every request**. Routes that don't take a tenant parameter use the auth header to resolve.
- **No layer-internal types in route responses**. Every response model lives in `api/schemas.py`. If a layer returns a private type, the route translates.
- **Auth is uniform**. Don't write a custom auth shim in any route.
